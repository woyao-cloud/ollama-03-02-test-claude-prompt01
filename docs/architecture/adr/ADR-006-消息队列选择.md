# ADR-006: 消息队列选择

## 状态
**已接受** (Accepted)

## 日期
2026-03-24

## 作者
系统架构师

---

## 上下文 (Context)

### 问题背景
系统需要选择消息队列方案，满足以下需求：

- 审计日志量大：1000万条/天，峰值50000条/分钟
- 登录接口需要异步写入日志，保证<100ms响应
- 支持事件驱动的业务逻辑
- 系统解耦，提高可扩展性

### 约束条件
根据 CONTEXT.md 中的 **Locked Decisions**：
- D-04: 使用Kafka 3+ 处理审计日志和异步事件（推荐但非强制）

---

### Kafka 配置

| 配置项 | 值 | 说明 |
|--------|-----|------|
| **Broker数量** | 3 | 高可用配置 |
| **Topic分区数** | 6 | audit-log topic |
| **副本因子** | 3 | 数据冗余 |
| **acks** | 1 | Producer确认级别 |
| **retries** | 3 | 发送失败重试 |
| **batch.size** | 16384 | 批量发送大小 |
| **linger.ms** | 5 | 等待批处理时间 |
| **compression.type** | snappy | 压缩算法 |

### Topic 设计

| Topic名称 | 分区数 | 保留期 | 消费者组 | 用途 |
|-----------|--------|--------|----------|------|
| audit-log | 6 | 7天 | audit-consumer | 审计日志入库 |
| user-events | 3 | 3天 | notification-service | 用户事件通知 |
| system-events | 3 | 1天 | monitoring-service | 系统监控 |
| dlq | 1 | 30天 | - | 死信队列 |

## 备选方案 (Alternatives)

### 备选1: RabbitMQ

**优点：**
- 消息路由灵活（Topic、Direct、Fanout）
- 管理界面友好
- 延迟队列原生支持
- 社区成熟，文档丰富

**缺点：**
- 吞吐量不如Kafka（单机万级 vs Kafka十万级）
- 持久化性能较差
- 集群模式复杂（镜像队列）
- 日志量大场景不如Kafka合适

**适用场景：**
- 实时性要求高（延迟敏感）
- 复杂路由需求
- 消息量不大的场景

**决策：** 拒绝。审计日志场景需要高吞吐，Kafka更合适。

### 备选2: 数据库异步表

**优点：**
- 无需引入新组件
- 实现简单
- 事务一致性容易保证

**缺点：**
- 数据库压力大
- 无法削峰填谷
- 消费端扩展性差
- 无法支持多消费者

**适用场景：**
- 消息量极小
- 简单异步任务

**决策：** 拒绝。1000万条/天的日志量不适合用数据库。

### 备选3: Redis Stream / List

**优点：**
- 无需额外组件（已有Redis）
- 实现简单
- 性能较好

**缺点：**
- 数据可靠性不如Kafka（内存存储）
- 无法支持多消费者组
- 消息堆积影响Redis性能
- 不支持分区扩展

**适用场景：**
- 消息量不大
- 简单队列场景

**决策：** 拒绝。审计日志需要高可靠性和扩展性。

## 后果 (Consequences)

### 正面后果

1. **高吞吐**：支持数十万条/分钟的消息处理
2. **持久化**：消息持久化到磁盘，可回放
3. **可扩展**：通过增加分区实现水平扩展
4. **多消费者**：支持多个消费者组独立消费
5. **削峰填谷**：消息堆积不影响Producer

### 负面后果

1. **运维复杂度**：需要维护Kafka集群
2. **延迟较高**：相比RabbitMQ延迟略高（ms级 vs 微秒级）
3. **学习成本**：Kafka概念较多（Partition、Offset、Consumer Group）

### 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| Kafka不可用 | 中 | 中 | 降级为同步写入DB，监控告警 |
| 消息丢失 | 低 | 高 | Producer acks=1，重试机制 |
| 消息重复 | 低 | 中 | Consumer幂等处理 |
| 消息堆积 | 中 | 中 | 监控告警，增加消费者 |
| 分区不均 | 低 | 中 | 合理设计Partition Key |

---

## 降级方案

### Kafka不可用时的降级策略

```
1. 检测Kafka连接失败
2. 切换到备用方案:
   ├─ 方案A: 同步写入DB（性能下降）
   ├─ 方案B: 写入本地日志文件
   └─ 方案C: 丢弃非关键日志
3. 记录告警
4. 定时重连Kafka
5. 恢复后切换回Kafka
```
