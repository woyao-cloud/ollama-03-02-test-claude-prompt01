# ADR-002: 认证方案

## 状态
**已接受** (Accepted)

## 日期
2026-03-24

## 作者
系统架构师

---

## 上下文 (Context)

### 问题背景
系统需要选择一种认证方案，满足以下需求：

- 支持1000万+用户的高并发认证
- 支持分布式部署（无状态）
- 安全性要求高（等保2.0三级）
- 支持单点登录（SSO）集成
- Token有效期：Access 15分钟，Refresh 7天

### 约束条件
根据 CONTEXT.md 中的 **Locked Decisions**：
- D-05: 采用 JWT（Access + Refresh Token）+ OAuth2.0 认证
- D-03: Redis必需（用于会话存储）

---

## 决策 (Decision)

### 最终选择的认证方案

**主方案：JWT + OAuth2.0 + Redis会话**

```
┌─────────────────────────────────────────────────────────────┐
│                      认证架构                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Client (Browser)                                          │
│       │                                                     │
│       │ POST /auth/login                                    │
│       │ {email, password}                                   │
│       ▼                                                     │
│   ┌───────────────────────────────────────┐                │
│   │           Auth Server                 │                │
│   │  ┌─────────────┐  ┌─────────────┐     │                │
│   │  │ 密码验证    │  │ 生成JWT     │     │                │
│   │  │ BCrypt      │  │ RSA256      │     │                │
│   │  └─────────────┘  └─────────────┘     │                │
│   └──────────┬────────────────────────────┘                │
│              │                                              │
│              ▼                                              │
│   ┌───────────────────────────────────────┐                │
│   │         Redis Cluster                 │                │
│   │  • Session存储                        │                │
│   │  • JWT黑名单                          │                │
│   │  • 登录失败计数                       │                │
│   └───────────────────────────────────────┘                │
│                                                             │
│   Response:                                                 │
│   {                                                         │
│     accessToken: "eyJ...",    // 15分钟有效期              │
│     refreshToken: "eyJ...",   // 7天有效期                 │
│     tokenType: "Bearer",                                   │
│     expiresIn: 900                                         │
│   }                                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 技术细节

| 组件 | 选择 | 配置 |
|------|------|------|
| **JWT签名算法** | RSA256 | 2048位RSA密钥对 |
| **Access Token** | JWT | 有效期15分钟 |
| **Refresh Token** | JWT | 有效期7天（记住我30天） |
| **Token存储** | Redis | Key: session:{userId}:{sessionId} |
| **会话管理** | Redis | 单用户最多5个同时在线会话 |
| **Token黑名单** | Redis | Key: jwt:blacklist:{tokenId} |

## 备选方案 (Alternatives)

### 备选1: Session + Cookie

**方案描述：**
- 服务端生成Session ID
- 存储在Cookie中
- Session数据存储在Redis

**优点：**
- 实现简单，兼容性好
- 服务端控制会话生命周期
- 随时可强制下线用户

**缺点：**
- 跨域支持复杂
- 移动端/App支持不佳
- 不支持微服务架构的无状态要求

**决策：** 拒绝。需要支持SSO和跨域场景，JWT更适合。

### 备选2: OAuth2 + OpenID Connect

**方案描述：**
- 使用Keycloak/Auth0等IDP
- 完整的OIDC流程
- 支持社交登录

**优点：**
- 企业级标准
- 支持SSO、MFA
- 社交登录集成简单

**缺点：**
- 引入额外组件，增加复杂度
- Keycloak资源占用高
- 自研需求为主，不需要多IDP

**决策：** 拒绝。v1.0采用自研JWT，v2.0可考虑引入OIDC。

### 备选3: JWT + 对称加密 (HS256)

**方案描述：**
- 使用HS256签名
- 密钥共享给所有服务

**优点：**
- 签名验证更快
- 实现更简单

**缺点：**
- 密钥泄露风险高
- 密钥轮换复杂
- 无法安全分发密钥到客户端

**决策：** 拒绝。使用RSA256非对称加密，私钥只保存在服务端。

---

## 后果 (Consequences)

### 正面后果

1. **无状态服务**：服务实例不保存会话，支持水平扩展
2. **性能好**：JWT验证无需查询数据库，纯内存计算
3. **安全性高**：RSA256非对称加密，私钥不外泄
4. **灵活性强**：支持多种客户端（Web、App、API）
5. **符合标准**：OAuth2.0标准，便于集成第三方

### 负面后果

1. **Token无法撤销**：有效期内只能加入黑名单
2. **Token体积较大**：比Session ID大
3. **密钥管理复杂**：RSA密钥对需要安全管理

### 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 私钥泄露 | 低 | 极高 | 密钥存储在K8s Secret，定期轮换 |
| Token伪造 | 低 | 高 | RSA256签名，定期更换密钥对 |
| Token劫持 | 中 | 高 | HTTPS传输，短有效期，IP绑定 |
| 暴力破解 | 中 | 中 | 登录限流，账户锁定策略 |

## 实现细节

### JWT Payload 结构

```json
{
  "sub": "user-uuid",
  "iss": "usermanagement",
  "aud": "usermanagement-api",
  "iat": 1711273200,
  "exp": 1711274100,
  "jti": "unique-token-id",
  "roles": ["ROLE_ADMIN", "ROLE_USER"],
  "permissions": ["user:create", "user:read"],
  "deptId": "dept-uuid"
}
```

### Redis 存储设计

```
# 会话存储
session:{userId}:{sessionId} → {
  accessTokenJti: "...",
  refreshTokenJti: "...",
  createdAt: "...",
  expiresAt: "...",
  ip: "...",
  userAgent: "..."
}

# JWT 黑名单
jwt:blacklist:{jti} → expired_time

# 登录失败计数
login:failed:{email} → count (TTL: 30min)
```
