## 2. 总体描述

### 2.1 系统愿景
构建一个企业级用户角色权限管理系统，支持1000万+用户规模，提供高效的RBAC权限管理、完整的审计追踪和高可用认证服务。

### 2.2 业务目标
1. 实现用户全生命周期管理（创建、启用、禁用、删除）
2. 建立灵活的RBAC权限模型（用户-角色-权限三级结构）
3. 提供细粒度权限控制（菜单、操作、字段、数据四级）
4. 建立完整的操作审计体系
5. 支撑高并发登录（10000 TPS）

### 2.3 用户角色定义

基于用户画像，系统支持以下角色：

| 角色ID | 角色名称 | 描述 | 典型用户 |
|--------|----------|------|----------|
| ROLE_SYS_ADMIN | 系统管理员 | 拥有所有权限，管理整个系统 | 张明（IT运维主管） |
| ROLE_DEPT_MANAGER | 部门经理 | 管理部门成员，审批权限申请 | 李华（销售总监） |
| ROLE_END_USER | 普通用户 | 使用系统功能，管理个人信息 | 王芳（产品经理） |
| ROLE_AUDITOR | 审计员 | 查看审计日志，生成报表 | 审计部门人员 |

### 2.4 系统边界

**系统内**: 用户管理、角色权限管理、审计日志、认证授权、系统配置
**系统外**: HR系统（用户同步）、邮件服务（通知）、短信服务（2FA）、企业微信/钉钉（SSO）

### 2.5 假设和依赖

**假设**:
- 用户已有基本的IT系统使用经验
- 企业有基本的IT基础设施（服务器、网络）
- 有明确的组织架构

**依赖**:
- PostgreSQL 15+ 数据库
- Redis 7+ 缓存（必需，用于会话和高并发）
- Kafka 3+ 消息队列（推荐，用于审计日志）
- SMTP 邮件服务器

## 4. 详细用例描述

### UC-01.01 用户CRUD

**用例ID**: UC-01.01
**用例名称**: 用户CRUD
**参与者**: 系统管理员
**前置条件**: 管理员已登录系统
**后置条件**: 用户信息更新成功
**触发事件**: 管理员需要管理用户

#### 主流程

**创建用户**:
1. 管理员进入用户管理页面
2. 点击"创建用户"按钮
3. 填写用户基本信息（姓名、邮箱、部门、手机号）
4. 选择用户角色（可多选）
5. 设置初始密码（或选择发送激活邮件）
6. 点击"保存"
7. 系统验证邮箱唯一性
8. 系统创建用户并加密存储密码
9. 系统发送激活邮件（如选择）
10. 系统记录创建日志
11. 系统显示创建成功

**查询用户**:
1. 管理员进入用户管理页面
2. 输入查询条件（姓名、邮箱、部门、状态）
3. 系统返回分页结果
4. 支持按创建时间、姓名排序

**更新用户**:
1. 管理员在用户列表选择用户
2. 点击"编辑"
3. 修改用户信息
4. 点击"保存"
5. 系统验证并更新
6. 系统记录更新日志

**删除用户**（软删除）:
1. 管理员选择用户
2. 点击"删除"
3. 系统确认提示
4. 系统标记用户为已删除
5. 系统注销用户会话
6. 系统记录删除日志

#### 业务规则
- BR-01.01.01: 邮箱必须唯一，格式正确
- BR-01.01.02: 手机号格式需符合国内标准
- BR-01.01.03: 创建用户时至少分配一个角色
- BR-01.01.04: 删除用户为软删除，保留数据用于审计
- BR-01.01.05: 不能删除系统预留的超级管理员

#### 输入字段

| 字段 | 类型 | 必填 | 校验规则 |
|------|------|------|----------|
| 姓名 | String | 是 | 2-50字符，中英文 |
| 邮箱 | String | 是 | 唯一，邮箱格式 |
| 手机号 | String | 否 | 11位手机号 |
| 部门 | String | 是 | 下拉选择 |
| 角色 | Array | 是 | 至少一个 |
| 状态 | Enum | 是 | ACTIVE/INACTIVE/PENDING |

### UC-01.05 部门管理

**用例ID**: UC-01.05
**用例名称**: 部门管理
**参与者**: 系统管理员
**前置条件**: 管理员已登录系统
**后置条件**: 部门信息更新成功
**触发事件**: 管理员需要管理组织架构

#### 主流程

**创建部门**:
1. 管理员进入组织架构页面
2. 点击"创建部门"按钮
3. 填写部门信息：
   - 部门名称（必填）
   - 部门编码（必填，唯一）
   - 上级部门（可选，树形选择）
   - 部门负责人（可选，选择用户）
   - 部门描述（可选）
   - 排序号（可选，默认0）
4. 点击"保存"
5. 系统验证部门编码唯一性
6. 系统验证上级部门存在且未形成循环依赖
7. 系统创建部门记录
8. 系统更新部门树缓存（Redis）
9. 系统记录创建日志
10. 系统显示创建成功

**查询部门**:
1. 管理员进入组织架构页面
2. 系统以树形结构展示部门层级
3. 支持展开/折叠子部门
4. 点击部门显示部门详情和成员列表
5. 支持按部门名称搜索

**更新部门**:
1. 管理员选择部门
2. 点击"编辑"
3. 修改部门信息
4. 点击"保存"
5. 系统验证（同创建）
6. 系统更新部门信息
7. 系统更新缓存
8. 如果部门负责人变更，更新用户部门信息
9. 系统记录更新日志

**删除部门**:
1. 管理员选择部门
2. 点击"删除"
3. 系统检查部门下是否有子部门
4. 系统检查部门下是否有用户
5. 如有子部门或用户，提示无法删除
6. 如无关联数据，确认删除
7. 系统标记部门为已删除（软删除）
8. 系统更新缓存
9. 系统记录删除日志

**调整部门层级**:
1. 管理员拖拽部门节点
2. 或点击"调整上级"选择新上级部门
3. 系统检查是否形成循环依赖（不能将自己或子部门设为上级）
4. 系统更新parent_id
5. 系统更新层级路径（path）
6. 系统更新所有子部门的层级路径
7. 系统更新缓存
8. 系统记录变更日志

#### 部门层级规则

**五级组织架构模型**:
```
公司（根节点，第1级）
├── 一级部门（中心/事业部，第2级）
│   ├── 二级部门（部门，第3级）
│   │   ├── 三级部门（组/科室，第4级）
│   │   │   ├── 四级部门（小组/分队，第5级）
│   │   │   └── 四级部门
│   │   └── 三级部门
│   └── 二级部门
└── 一级部门
```

**约束**:
- 最多支持 5 级部门层级
- 部门编码全局唯一，建议格式：DEPT-001
- 每个部门必须有上级（根节点除外）
- 部门删除前必须清空子部门和用户

#### 业务规则
- BR-01.05.01: 部门编码必须全局唯一，建议使用前缀DEPT-
- BR-01.05.02: 部门层级最多5级
- BR-01.05.03: 删除部门前必须确保无子部门且无用户归属
- BR-01.05.04: 调整部门层级时不能形成循环依赖
- BR-01.05.05: 修改部门会影响该部门及子部门用户的数据权限
- BR-01.05.06: 部门变更需记录审计日志
- BR-01.05.07: 部门信息变更后需刷新Redis缓存

#### 部门与用户关系
- 每个用户必须属于一个部门（多对一）
- 部门负责人必须是该部门或上级部门的用户（可选）
- 用户调部门时，数据权限自动变更到新部门
- 支持查看部门成员列表

#### 输入字段

| 字段 | 类型 | 必填 | 校验规则 |
|------|------|------|----------|
| 部门名称 | String | 是 | 2-100字符 |
| 部门编码 | String | 是 | 唯一，英文数字下划线 |
| 上级部门 | UUID | 否 | 引用department表 |
| 部门负责人 | UUID | 否 | 引用user表 |
| 部门描述 | String | 否 | 最大500字符 |
| 排序号 | Int | 否 | 默认0，越大越靠前 |

### UC-02.02 权限定义与管理

**用例ID**: UC-02.02
**用例名称**: 权限定义与管理
**参与者**: 系统管理员
**前置条件**: 管理员已登录
**后置条件**: 权限配置更新
**触发事件**: 需要配置或调整权限

#### 主流程

1. 管理员进入权限管理页面
2. 系统显示权限树形结构
3. 管理员可执行以下操作：
   - 查看现有权限
   - 创建新权限
   - 编辑权限
   - 删除未使用权限

#### 权限四级模型

**一级: 菜单权限**
- 控制用户可见的菜单项
- 粒度: 模块级别
- 示例: "用户管理"菜单可见性

**二级: 操作权限**
- 控制页面上的按钮操作
- 粒度: 功能级别
- 示例: "创建用户"、"编辑用户"、"删除用户"按钮

**三级: 字段权限**
- 控制字段的显示/编辑
- 粒度: 字段级别
- 示例: "手机号"字段可查看但不可编辑

**四级: 数据权限**
- 控制可见的数据范围
- 粒度: 数据行级别
- 范围: 全部/本部门/本人
- **本部门范围说明**: 包含用户所属部门及其所有子部门的数据

#### 业务规则
- BR-02.02.01: 权限代码必须唯一，使用点号分隔（如：user:create）
- BR-02.02.02: 删除权限前需检查是否被角色引用
- BR-02.02.03: 权限变更后已登录用户需重新登录生效（或15分钟内生效）

### UC-03.01 JWT认证

**用例ID**: UC-03.01
**用例名称**: JWT认证
**参与者**: 普通用户、系统
**前置条件**: 用户已注册
**后置条件**: 用户获得访问令牌
**触发事件**: 用户尝试登录

#### 主流程

1. 用户输入邮箱和密码
2. 用户点击"登录"
3. 系统验证邮箱和密码（BCrypt比对）
4. 系统检查账户状态（是否锁定、是否激活）
5. 系统检查登录IP是否在白名单（如配置）
6. 系统生成Access Token（JWT，有效期15分钟）
7. 系统生成Refresh Token（有效期7天）
8. 系统将Token存入Redis（支持分布式会话）
9. 系统记录登录日志（异步写入Kafka）
10. 系统返回Token给客户端
11. 客户端存储Token（HttpOnly Cookie + LocalStorage双存储）

#### 备选流程

**密码错误**:
1. 密码比对失败
2. 系统增加失败计数（Redis记录）
3. 如果失败次数 >= 5次，锁定账户30分钟
4. 返回"用户名或密码错误"

**账户未激活**:
1. 检查用户状态为PENDING
2. 返回"账户未激活，请查收邮件"

**账户锁定**:
1. 检查Redis中失败次数 >= 5
2. 返回"账户已锁定，请30分钟后重试或联系管理员"

#### 业务规则
- BR-03.01.01: Access Token有效期15分钟，Refresh Token 7天
- BR-03.01.02: Token使用RSA256签名，私钥服务端保存
- BR-03.01.03: 登录失败5次锁定30分钟
- BR-03.01.04: 支持"记住我"功能（Refresh Token延长至30天）
- BR-03.01.05: 单用户最多5个同时在线会话

#### 性能要求
- 登录接口响应 < 100ms
- 支持 10,000 TPS（通过Redis缓存和异步日志）

### UC-04.01 操作日志记录

**用例ID**: UC-04.01
**用例名称**: 操作日志记录
**用例类型**: 系统用例（自动执行）
**触发事件**: 用户执行敏感操作

#### 主流程

1. 用户执行敏感操作（登录、登出、创建/修改/删除用户、权限变更等）
2. AOP拦截器捕获操作
3. 系统收集日志信息：
   - 操作时间（精确到毫秒）
   - 操作用户ID
   - 操作类型（CREATE/UPDATE/DELETE/LOGIN/LOGOUT等）
   - 操作对象（表名+记录ID）
   - 操作前数据（JSON格式）
   - 操作后数据（JSON格式）
   - 操作结果（成功/失败）
   - 客户端IP
   - 浏览器User-Agent
   - 会话ID
4. 系统将日志写入Kafka（异步）
5. 消费者服务消费日志写入PostgreSQL

#### 需记录的敏感操作

| 操作类型 | 说明 |
|----------|------|
| USER_CREATE | 创建用户 |
| USER_UPDATE | 更新用户 |
| USER_DELETE | 删除用户 |
| USER_LOGIN | 用户登录 |
| USER_LOGOUT | 用户登出 |
| ROLE_ASSIGN | 分配角色 |
| PERMISSION_CHANGE | 权限变更 |
| PASSWORD_RESET | 密码重置 |
| SYSTEM_CONFIG | 系统配置变更 |

### 7.2 部门实体

| 字段 | 类型 | 描述 | 约束 |
|------|------|------|------|
| id | UUID | 主键 | PK |
| name | VARCHAR(100) | 部门名称 | NOT NULL |
| code | VARCHAR(50) | 部门编码（如DEPT-001） | UK, NOT NULL |
| parent_id | UUID | 父部门ID | FK → department.id, 根节点为NULL |
| manager_id | UUID | 部门负责人ID | FK → user.id |
| level | INT | 层级（1-5，1为公司） | NOT NULL, DEFAULT 1 |
| path | VARCHAR(500) | 层级路径（如/1/2/5） | NOT NULL, 用于快速查询子树 |
| sort_order | INT | 排序号 | DEFAULT 0 |
| status | ENUM | 状态（ACTIVE/INACTIVE） | NOT NULL, DEFAULT ACTIVE |
| created_at | TIMESTAMP | 创建时间 | NOT NULL |
| updated_at | TIMESTAMP | 更新时间 | NOT NULL |
| deleted_at | TIMESTAMP | 删除时间（软删除） | - |

**部门实体说明**:
- 自关联结构：parent_id 指向同一表的父部门，支持无限层级（业务限制最多5级）
- path 字段：存储从根到当前节点的路径，便于快速查询所有子部门
- level 字段：表示部门层级，1=公司，2=一级部门，3=二级部门...
- 删除约束：存在子部门或归属用户时不可删除
