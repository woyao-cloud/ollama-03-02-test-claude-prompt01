# ADR-007: 部门树形结构设计

## 状态
**已接受** (Accepted)

## 日期
2026-03-24

## 作者
系统架构师

---

## 上下文 (Context)

### 问题背景
系统需要支持部门树形结构管理，满足以下需求：

- 最多5级部门层级（公司->部门->子部门）
- 支持递归查询子部门
- 部门层级变更时数据权限自动更新
- 高性能查询（支持1000万用户场景）

### 约束条件
根据 FRD 中的需求：
- 部门层级最多5级
- 需要支持快速查询子部门
- 部门变更需要更新所有子部门

---

## 决策 (Decision)

### 最终选择：Materialized Path (材质化路径) + Level 字段

```
┌─────────────────────────────────────────────────────────────┐
│                Materialized Path 模式                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   公司                                                       │
│   ├── 研发中心 (path: /1, level: 2)                          │
│   │   ├── 后端组 (path: /1/2, level: 3)                      │
│   │   │   ├── Java组 (path: /1/2/5, level: 4)               │
│   │   │   └── Go组 (path: /1/2/6, level: 4)                 │
│   │   └── 前端组 (path: /1/3, level: 3)                      │
│   └── 销售中心 (path: /4, level: 2)                          │
│       └── 华东区 (path: /4/7, level: 3)                      │
│                                                             │
│   表结构:                                                   │
│   ┌────┬──────────┬──────────┬───────┬────────────────┐    │
│   │ id │   name   │ parent_id│ level │     path       │    │
│   ├────┼──────────┼──────────┼───────┼────────────────┤    │
│   │ 1  │研发中心  │ NULL     │   2   │ /1             │    │
│   │ 2  │后端组    │ 1        │   3   │ /1/2           │    │
│   │ 5  │Java组    │ 2        │   4   │ /1/2/5         │    │
│   │ 6  │Go组      │ 2        │   4   │ /1/2/6         │    │
│   │ 3  │前端组    │ 1        │   3   │ /1/3           │    │
│   │ 4  │销售中心  │ NULL     │   2   │ /4             │    │
│   │ 7  │华东区    │ 4        │   3   │ /4/7           │    │
│   └────┴──────────┴──────────┴───────┴────────────────┘    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 核心设计

| 字段 | 类型 | 用途 |
|------|------|------|
| **id** | UUID | 主键 |
| **parent_id** | UUID | 父部门ID（自关联） |
| **level** | INT | 部门层级（1-5） |
| **path** | VARCHAR(500) | 材质化路径（/1/2/5） |

### 查询示例

```sql
-- 查询某部门的所有子部门（高效）
SELECT * FROM department WHERE path LIKE '/1/2%';

-- 查询某部门的所有上级（获取path后拆分）
-- path = '/1/2/5' → 查询 id IN (1, 2, 5)

-- 查询同级部门
SELECT * FROM department WHERE parent_id = 2;

-- 查询某层级所有部门
SELECT * FROM department WHERE level = 3;
```

### 更新流程

```
部门层级变更时的处理流程:

1. 检查是否形成循环依赖
2. 更新 parent_id
3. 计算新的 level = 父level + 1
4. 计算新的 path = 父path + '/' + id
5. 递归更新所有子部门的 level 和 path
6. 更新Redis缓存
7. 记录审计日志
```

## 备选方案 (Alternatives)

### 备选1: 邻接表模式（Adjacency List）

**方案描述：**
- 只使用 `parent_id` 字段
- 查询子部门需要递归或使用CTE

**优点：**
- 结构简单
- 插入更新容易
- 支持无限层级

**缺点：**
- 查询子部门需要递归（性能差）
- 查询所有上级需要递归
- 不适合大数据量

**SQL示例：**
```sql
-- 查询子部门（需要递归CTE）
WITH RECURSIVE dept_tree AS (
    SELECT id, name, parent_id, 1 as depth
    FROM department WHERE id = 1
    UNION ALL
    SELECT d.id, d.name, d.parent_id, dt.depth + 1
    FROM department d
    JOIN dept_tree dt ON d.parent_id = dt.id
)
SELECT * FROM dept_tree;
```

**决策：** 拒绝。递归查询在大数据量场景性能差。

### 备选2: 嵌套集模式（Nested Set）

**方案描述：**
- 使用 `left` 和 `right` 字段表示节点范围
- 查询子树无需递归

**优点：**
- 查询子树效率高（范围查询）
- 查询所有子节点方便

**缺点：**
- 插入更新复杂度高（需要更新多个节点）
- 不适合频繁变更的树
- 层级深度有限制（数字范围）

**SQL示例：**
```sql
-- 表结构
CREATE TABLE department (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    left INT,
    right INT
);

-- 查询子部门
SELECT * FROM department WHERE left > 2 AND right < 11;
```

**决策：** 拒绝。部门变更是常见操作，嵌套集更新成本太高。

### 备选3: 闭包表模式（Closure Table）

**方案描述：**
- 额外创建一张关系表存储所有祖先-后代关系
- 查询子部门或祖先都很高效

**优点：**
- 查询祖先和后代都高效
- 支持复杂的树操作

**缺点：**
- 需要维护额外表
- 插入更新复杂（需要维护多行关系）
- 存储空间占用大

**SQL示例：**
```sql
-- 部门表
CREATE TABLE department (id INT PRIMARY KEY, name VARCHAR(100));

-- 闭包表
CREATE TABLE department_closure (
    ancestor_id INT REFERENCES department(id),
    descendant_id INT REFERENCES department(id),
    depth INT,
    PRIMARY KEY (ancestor_id, descendant_id)
);

-- 查询子部门
SELECT d.* FROM department d
JOIN department_closure c ON d.id = c.descendant_id
WHERE c.ancestor_id = 1;
```

**决策：** 拒绝。实现复杂度高，5级部门场景Materialized Path足够。

## 后果 (Consequences)

### 正面后果

1. **查询高效**：查询子部门使用LIKE索引，O(log n)复杂度
2. **更新简单**：只更新当前节点和子节点的path
3. **层级限制**：level字段天然支持5级限制
4. **可读性好**：path字段直观展示层级关系

### 负面后果

1. **path长度限制**：需要足够长的字段存储路径
2. **更新子树**：需要递归更新子节点的path
3. **数字ID依赖**：path依赖自增ID或固定ID

### 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| path字段溢出 | 低 | 高 | 使用500字符长度，足够存储10+层级 |
| 更新不一致 | 中 | 高 | 事务包裹，异常回滚 |
| 循环依赖 | 低 | 高 | 应用层检查，数据库约束 |
| 并发更新 | 中 | 中 | 乐观锁控制 |

## 性能对比

| 操作 | 邻接表 | 嵌套集 | 闭包表 | Materialized Path |
|------|--------|--------|--------|-------------------|
| 查询子树 | 慢(递归) | 快 | 快 | 快 |
| 查询祖先 | 慢(递归) | 慢 | 快 | 快(拆分path) |
| 查询同级 | 快 | 快 | 快 | 快 |
| 插入节点 | 快 | 慢(更新多行) | 慢(更新多行) | 快 |
| 删除节点 | 快 | 慢(更新多行) | 慢(更新多行) | 快 |
| 移动节点 | 快 | 慢(重建树) | 慢(更新多行) | 中等(更新子树) |
| 存储空间 | 小 | 小 | 大 | 小 |

## Redis缓存设计

```
# 部门树缓存
department:tree → 完整树JSON
TTL: 30分钟

# 部门子节点列表（直接子节点）
department:{id}:children → [deptId1, deptId2, ...]
TTL: 30分钟

# 部门信息
department:{id}:info → JSON
TTL: 30分钟

# 部门路径缓存（用于快速查询上级）
department:{id}:path → /1/2/5
TTL: 30分钟
```
