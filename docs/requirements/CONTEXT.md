## 用户决策

### 已锁定决策（Locked）

决策是强制性的，必须严格遵循。

#### D-01: 技术栈选择 - Spring Boot + Next.js
- **决策**: 后端采用 Spring Boot 3.5 + JDK 21，前端采用 Next.js 16
- **原因**:
  - 团队熟悉 Java 和 Spring 生态
  - Next.js 16 支持 Server Components，性能好
  - 类型安全（Java + TypeScript）
- **影响**:
  - 需要 JDK 21 环境
  - 需要 Node.js 20+ 环境
  - 前后端分离架构
- **状态**: LOCKED

#### D-02: 数据库选择 - PostgreSQL
- **决策**: 采用 PostgreSQL 15 作为关系型数据库
- **原因**:
  - 开源、稳定、性能优秀
  - 支持 JSONB、UUID 等特性
  - 良好的 Flyway 支持
- **影响**:
  - 需要 PostgreSQL 15+ 服务器
  - 使用 Flyway 进行版本管理
- **状态**: LOCKED

#### D-03: 缓存方案 - Redis（必需）
- **决策**: 必须使用 Redis 7+ 作为分布式缓存和会话存储
- **原因**:
  - 支持 1000万用户规模的高并发登录
  - 登录模块需要 10000 TPS 支持
  - 分布式会话管理需要
- **影响**:
  - 架构设计必须包含 Redis 集群
  - 登录流程依赖 Redis
  - 限流、缓存、JWT黑名单都使用 Redis
- **状态**: LOCKED

#### D-04: 消息队列 - Kafka（推荐但非强制）
- **决策**: 使用 Kafka 3+ 处理审计日志和异步事件
- **原因**:
  - 审计日志量大（1000万条/天）
  - 异步写入降低登录延迟
  - 支持事件流分析
- **影响**:
  - **推荐但非强制**（初期可用数据库替代）
  - 日志模块设计需要考虑 Kafka
- **状态**: LOCKED

#### D-05: 认证方式 - JWT + OAuth2
- **决策**: 采用 JWT（Access + Refresh Token）+ OAuth2.0 认证
- **原因**:
  - 无状态，适合分布式部署
  - 支持 SSO 集成
  - 业界标准方案
- **影响**:
  - Access Token 15分钟有效期
  - Refresh Token 7天有效期
  - 需要 Redis 存储会话
- **状态**: LOCKED

#### D-06: 权限模型 - RBAC 四级权限
- **决策**: 采用 RBAC 模型，支持四级权限粒度
- **原因**:
  - 菜单级 + 操作级 + 字段级 + 数据级
  - 灵活且强大的权限控制
  - 满足企业级需求
- **影响**:
  - 权限表设计复杂
  - 需要缓存权限信息
- **状态**: LOCKED

#### D-07: 用户规模 - 1000万注册用户
- **决策**: 系统必须支持 1000万+ 注册用户
- **原因**:
  - 业务需求为外部访问用户总量
  - 需要高并发架构支持
- **影响**:
  - 数据库分片预留
  - 缓存容量规划
  - 水平扩展设计
- **状态**: LOCKED

#### D-08: 性能目标 - 登录 10000 TPS
- **决策**: 登录模块必须支持 10000 TPS
- **原因**:
  - 高并发场景需求
  - 峰值 50000 并发用户
- **影响**:
  - 必须使用 Redis
  - 审计日志必须异步
  - 需要负载均衡
- **状态**: LOCKED

#### D-09: 测试覆盖率 - 后端 >85%
- **决策**: 后端单元测试覆盖率必须 >= 85%
- **原因**:
  - 项目质量门禁要求
  - 保障系统稳定性
- **影响**:
  - 需要编写大量测试代码
  - 使用 JaCoCo 监控
- **状态**: LOCKED

#### D-10: 部署环境 - Docker + Kubernetes
- **决策**: 使用 Docker 容器化 + Kubernetes 编排
- **原因**:
  - 标准化部署
  - 支持自动扩缩容
  - 云原生架构
- **影响**:
  - 需要 K8s 集群
  - 需要编写 Dockerfile、K8s YAML
- **状态**: LOCKED

### 延迟决策（Deferred）

决策推迟到后续版本或阶段。

#### DF-01: LDAP/AD 集成
- **决策**: LDAP/AD 企业目录集成
- **延迟原因**: v1.0 先支持基础认证，v1.2 再集成 LDAP
- **预计解决时间**: v1.2 (2026-06-30)

#### DF-02: 双因素认证（强制）
- **决策**: 强制要求所有用户启用 2FA
- **延迟原因**: v1.0 作为可选功能，v1.1 支持强制启用
- **预计解决时间**: v1.1 (2026-05-31)

#### DF-03: AI 异常检测
- **决策**: 使用 AI 检测登录异常行为
- **延迟原因**: 需要训练数据，v2.0 再引入
- **预计解决时间**: v2.0 (2026-09-30)

#### DF-04: 多租户支持
- **决策**: SaaS 化多租户架构
- **延迟原因**: 当前为单租户，未来版本考虑
- **预计解决时间**: 2026 Q4

#### DF-05: 移动端 App
- **决策**: 原生移动端 App
- **延迟原因**: v1.0 使用响应式 Web，原生 App 后续评估
- **预计解决时间**: 待定
