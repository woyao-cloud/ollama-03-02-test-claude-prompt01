# ADR-005: 缓存策略

## 状态
**已接受** (Accepted)

## 日期
2026-03-24

## 作者
系统架构师

---

## 上下文 (Context)

### 问题背景
系统需要设计缓存策略，满足以下需求：

- 登录接口支持10000 TPS，响应时间 < 100ms
- 用户权限查询需要快速响应
- 部门树查询需要支持递归
- 系统需要无状态，支持水平扩展
- 缓存数据需要与数据库保持一致

### 约束条件
根据 CONTEXT.md 中的 **Locked Decisions**：
- D-03: Redis 7+ 必需（分布式缓存和会话存储）

---

## 决策 (Decision)

### 最终缓存架构

```
┌─────────────────────────────────────────────────────────────┐
│                    多级缓存架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   L1: 本地缓存 (Caffeine)                                   │
│   ├─ 存储: 热点数据、系统配置                                │
│   ├─ TTL: 5分钟                                             │
│   ├─ 策略: LRU淘汰                                          │
│   └─ 容量: 最多10,000条                                     │
│                                                             │
│   L2: 分布式缓存 (Redis Cluster)                             │
│   ├─ 存储: 用户权限、会话、部门树                            │
│   ├─ TTL: 15分钟 - 7天                                      │
│   ├─ 策略: 主动更新 + 过期删除                               │
│   └─ 集群: 6节点（3主3从）                                   │
│                                                             │
│   L3: 持久化存储 (PostgreSQL)                                │
│   ├─ 存储: 全量数据                                         │
│   └─ 策略: 永久存储                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 缓存策略表

| 数据类型 | L1缓存 | L2缓存 | TTL | 更新策略 |
|----------|--------|--------|-----|----------|
| 用户权限 | Caffeine | Redis | 15分钟 | 主动更新 |
| 用户会话 | - | Redis | 15分钟/7天 | 事件驱动 |
| 部门树 | Caffeine | Redis | 30分钟 | 事件驱动 |
| 系统配置 | Caffeine | Redis | 1小时 | 定时刷新 |
| JWT黑名单 | - | Redis | Token过期时间 | 写入时设置 |
| 登录失败计数 | - | Redis | 30分钟 | 累加 |
| 限流计数 | - | Redis | 1分钟 | 累加 |

## Redis Key 设计

### 命名空间规范

**统一前缀格式**: `um:{module}:{resource}:{identifier}`

| 组成部分 | 说明 | 示例 |
|----------|------|------|
| `um` | 应用前缀 (UserManagement) | - |
| `{module}` | 模块名 | `auth`, `user`, `dept`, `sys` |
| `{resource}` | 资源类型 | `session`, `perms`, `tree` |
| `{identifier}` | 唯一标识 | `{userId}`, `{deptId}` |

### 用户相关

```
# 用户会话
um:auth:session:{userId}:{sessionId} → {
    "accessTokenJti": "...",
    "refreshTokenJti": "...",
    "createdAt": "...",
    "expiresAt": "...",
    "ip": "...",
    "userAgent": "..."
}
TTL: 7天

# 用户权限缓存
um:user:perms:{userId} → Set["user:create", "user:read", ...]
TTL: 15分钟

# 用户角色缓存
um:user:roles:{userId} → Set["role-uuid-1", "role-uuid-2"]
TTL: 15分钟

# 用户信息缓存（可选）
um:user:info:{userId} → JSON
TTL: 15分钟
```

### 安全相关

```
# JWT黑名单
um:auth:blacklist:{jti} → {expiredAt: "..."}
TTL: Token剩余有效期

# 登录失败计数
um:auth:failed:{email} → count
TTL: 30分钟

# 账户锁定
um:auth:locked:{email} → lockedUntil
TTL: 锁定剩余时间

# 限流计数
um:sys:ratelimit:{ip}:{path} → count
TTL: 1分钟

# IP白名单
um:user:whitelist:{userId} → Set["192.168.1.1", ...]
TTL: 1小时
```

### 配置相关

```
# 系统配置
um:sys:config:{key} → value
TTL: 1小时

# 密码策略
um:sys:password:policy → JSON
TTL: 1小时

# 部门树
um:dept:tree → JSON
TTL: 30分钟

# 部门子节点
um:dept:children:{id} → List[deptId]
TTL: 30分钟

# 部门路径
um:dept:path:{id} → "/1/2/5"
TTL: 30分钟
```

## 缓存一致性策略

### Cache Aside 模式

```
┌─────────────────────────────────────────────────────────────┐
│                    Cache Aside 模式                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  读操作:                                                    │
│  ┌────────┐    ┌────────┐    ┌────────┐                    │
│  │ Client │───►│ Cache  │───►│   DB   │                    │
│  │        │    │ 命中？  │    │        │                    │
│  └────────┘    └───┬────┘    └────┬───┘                    │
│                    │              │                        │
│                    │命中          │未命中                   │
│                    │              │                        │
│                    ▼              ▼                        │
│               返回数据      写入Cache                      │
│                               返回数据                     │
│                                                             │
│  写操作:                                                    │
│  ┌────────┐    ┌────────┐    ┌────────┐                    │
│  │ Client │───►│   DB   │───►│ Cache  │                    │
│  │        │    │ 更新    │    │ 删除   │                    │
│  └────────┘    └────────┘    └────────┘                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 缓存更新策略

| 场景 | 策略 | 实现 |
|------|------|------|
| 用户权限变更 | 主动失效 | 更新DB后删除Redis缓存 |
| 部门变更 | 消息通知 | 发送事件，各节点清理本地缓存 |
| 配置变更 | 定时刷新 | 1小时自动过期 |
| 热点数据 | 预加载 | 启动时加载到本地缓存 |

## 备选方案 (Alternatives)

### 备选1: 只使用Redis，无本地缓存

**优点：**
- 实现简单
- 数据一致性容易保证

**缺点：**
- 网络RTT增加延迟
- Redis压力大
- 无法应对热点数据突发流量

**决策：** 拒绝。本地缓存是高性能的必要手段。

### 备选2: Write Through 模式

**优点：**
- 缓存和DB同步更新
- 数据一致性高

**缺点：**
- 写操作延迟增加
- 代码复杂度增加

**决策：** 拒绝。Cache Aside在大多数场景足够，且性能更好。

### 备选3: Write Behind 模式

**优点：**
- 写操作响应快
- 批量写入DB效率高

**缺点：**
- 数据丢失风险
- 实现复杂度高
- 不适合权限数据

**决策：** 拒绝。权限数据要求高一致性，不适合异步写入。

## 后果 (Consequences)

### 正面后果

1. **高性能**：本地缓存命中时 < 5ms响应
2. **高可用**：Redis Cluster支持故障自动转移
3. **可扩展**：无状态服务支持水平扩展
4. **低成本**：缓存减少数据库压力

### 负面后果

1. **一致性风险**：缓存与DB可能短暂不一致
2. **复杂度增加**：需要处理缓存穿透、击穿、雪崩
3. **内存成本**：需要更多内存存储缓存数据

### 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 缓存穿透 | 中 | 高 | 布隆过滤器，空值缓存 |
| 缓存击穿 | 中 | 高 | 热点数据永不过期，互斥锁 |
| 缓存雪崩 | 低 | 极高 | 随机TTL，多级缓存 |
| Redis故障 | 中 | 高 | Cluster模式，降级到DB |

---

## 缓存降级策略

```
Redis不可用时的降级流程:

1. 检测Redis连接失败
2. 切换到直连数据库模式
3. 本地缓存继续生效
4. 记录告警日志
5. 定时重连Redis
6. 恢复后自动切换回缓存模式
```
