# ADR-003: 高并发架构

## 状态
**已接受** (Accepted)

## 日期
2026-03-24

## 作者
系统架构师

## 上下文 (Context)

### 问题背景
系统需要支持以下高并发场景：

- **用户规模**：1000万注册用户，100万日活用户
- **登录性能**：支持10000 TPS，响应时间 < 100ms
- **并发用户**：10000+ 同时在线，峰值50000
- **日志写入**：1000万条/天，峰值50000条/分钟

### 约束条件
根据 CONTEXT.md 中的 **Locked Decisions**：
- D-08: 登录模块必须支持10000 TPS
- D-03: Redis必需（用于会话和高并发）
- D-04: Kafka推荐（用于审计日志）

### 关键设计决策

| 设计点 | 决策 | 理由 |
|--------|------|------|
| **服务状态** | 无状态 | 支持水平扩展，K8s友好 |
| **会话存储** | Redis Cluster | 分布式共享，高性能 |
| **审计日志** | Kafka异步写入 | 削峰填谷，不阻塞主流程 |
| **数据库** | 主从 + 读写分离 | 读多写少场景优化 |
| **缓存** | 多级缓存 | 本地缓存+Redis，减少RTT |
| **限流** | 网关+应用层双限流 | 分层防护 |

## 备选方案 (Alternatives)

### 备选1: 有状态服务 + Session粘滞

**方案描述：**
- 服务实例保存Session
- 负载均衡使用Session粘滞（IP Hash）

**优点：**
- 不依赖外部存储
- 实现简单

**缺点：**
- 不支持弹性扩缩容
- 实例故障导致Session丢失
- 负载可能不均衡

**决策：** 拒绝。需要支持自动扩缩容和故障转移。

### 备选2: 单体架构 + 垂直扩展

**方案描述：**
- 单个大实例部署
- 增加CPU/内存提升性能

**优点：**
- 架构简单
- 没有分布式复杂度

**缺点：**
- 扩展有上限（单机器限制）
- 单点故障风险高
- 成本高昂

**决策：** 拒绝。无法满足10000 TPS的长期扩展需求。

### 备选3: 数据库直连Session

**方案描述：**
- Session存储在PostgreSQL
- 定期清理过期Session

**优点：**
- 减少组件数量
- Session持久化

**缺点：**
- 数据库压力大
- 性能不如Redis
- 无法支持10000 TPS

**决策：** 拒绝。登录高并发场景需要Redis的高性能。

## 后果 (Consequences)

### 正面后果

1. **高可用性**：多实例部署，单点故障自动切换
2. **弹性扩展**：K8s HPA根据负载自动扩缩容
3. **高性能**：Redis缓存减少数据库压力
4. **削峰填谷**：Kafka缓冲审计日志写入
5. **无状态服务**：易于管理，快速启动

### 负面后果

1. **架构复杂度**：引入Redis、Kafka增加运维复杂度
2. **数据一致性**：缓存与数据库需要保持一致
3. **网络延迟**：服务间调用增加网络开销
4. **故障排查**：分布式系统问题定位更复杂

### 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| Redis故障 | 中 | 高 | Redis Cluster多主从，自动故障转移 |
| Kafka故障 | 中 | 中 | 降级为同步写入，监控告警 |
| DB连接池耗尽 | 中 | 高 | HikariCP调优，监控告警 |
| 缓存击穿 | 低 | 高 | 布隆过滤器，热点数据预加载 |
| 雪崩效应 | 中 | 高 | 熔断降级，限流，超时设置 |

## 性能优化措施

### 登录接口优化 (< 100ms)

```
┌─────────────────────────────────────────────────────────────┐
│                    登录性能优化                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 数据库连接池优化                                         │
│     • HikariCP: max pool size = 20                          │
│     • connection timeout = 5s                               │
│     • keepalive = true                                      │
│                                                             │
```

### 数据库连接池配置 (HikariCP)

**环境差异化配置**:

```yaml
# application-dev.yml (开发环境)
spring:
  datasource:
    hikari:
      maximum-pool-size: 10
      minimum-idle: 2
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
      leak-detection-threshold: 0  # 开发环境关闭

# application-test.yml (测试环境)
spring:
  datasource:
    hikari:
      maximum-pool-size: 15
      minimum-idle: 5
      connection-timeout: 20000
      idle-timeout: 300000
      max-lifetime: 1200000
      leak-detection-threshold: 30000

# application-prod.yml (生产环境)
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 10
      connection-timeout: 5000
      idle-timeout: 300000
      max-lifetime: 1200000
      leak-detection-threshold: 60000
      connection-test-query: SELECT 1
      validation-timeout: 3000
```

**配置说明**:

| 配置项 | 开发环境 | 测试环境 | 生产环境 | 说明 |
|--------|----------|----------|----------|------|
| maximum-pool-size | 10 | 15 | 20 | 最大连接数 = (核心数 * 2) + 有效磁盘数 |
| minimum-idle | 2 | 5 | 10 | 最小空闲连接数 |
| connection-timeout | 30s | 20s | 5s | 连接获取超时时间 |
| idle-timeout | 10min | 5min | 5min | 空闲连接超时回收时间 |
| max-lifetime | 30min | 20min | 20min | 连接最大生命周期（需小于数据库超时） |
| leak-detection-threshold | 关闭 | 30s | 60s | 连接泄露检测阈值 |

### 缓存策略

| 数据类型 | 缓存位置 | TTL | 更新策略 |
|----------|----------|-----|----------|
| 用户权限 | Redis + 本地 | 15分钟 | 主动更新 |
| 部门树 | Redis | 30分钟 | 事件驱动 |
| 系统配置 | Redis + 本地 | 1小时 | 定时刷新 |
| JWT黑名单 | Redis | Token过期时间 | 写入时设置 |
